# Phase 03: 排行榜基础下载 - Context

**Gathered:** 2026-02-25
**Status:** Ready for planning

<domain>
## Phase Boundary

实现从 pixiv 下载每日排行榜内容的核心能力,包括获取排行榜数据、下载图片文件、避免速率限制。这个阶段交付最小可用的下载功能,为后续的元数据扩展、多排行榜支持、用户体验优化奠定基础。

</domain>

<decisions>
## Implementation Decisions

### 下载行为
- **CLI 命令**: `pixiv-downloader download` (子命令模式,与 login/status/refresh 平级)
- **输出格式**: 默认 JSON 输出 (第三方友好,与 refresh 命令风格一致)
  - JSON 包含: 下载统计、成功列表、失败列表、文件路径
  - 不输出 Rich 进度条 (除非添加 --human 参数)
- **日期参数**: `--date 2026-02-23` (默认今天)
  - 格式: YYYY-MM-DD
  - 支持今天、昨天、指定日期

### 文件组织
- **保存路径**: 可配置,默认固定目录
  - 默认: `./pixiv-downloads/` (可通过 --output 覆盖)
- **文件命名**: `作品ID_标题.扩展名`
  - 示例: `12345678_美丽的风景.jpg`
  - 标题中的特殊字符需清理 (Windows 兼容)
- **目录结构**: `排行榜名称-日期/作品ID_标题.扩展名`
  - 示例: `daily-2026-02-23/12345678_美丽的风景.jpg`
  - 通用原则: 根据下载类型组织文件夹
    - 排行榜: `排行榜名称-日期/`
    - 作品集: `作品集ID/`
    - 画师: `画师名称/`

### 速率控制
- **请求间隔**: 保守策略 (大间隔+随机抖动)
  - 固定基础间隔: 2-3 秒
  - 随机抖动: ±0.5-1.5 秒
  - 避免 pixiv 反爬虫机制
- **并发下载**: 串行下载 (一次一张)
  - 不使用多线程/异步并发
  - 简化错误处理和状态管理

### 错误处理
- **单图失败**: 跳过并继续
  - 记录失败到 JSON 输出
  - 包含失败原因 (网络错误、404、权限等)
  - 第三方程序可根据失败列表重试
- **断点续传**: Phase 3 不支持,延后到 Phase 7
  - 重复运行会重新下载所有图片
  - Phase 7 实现增量下载和文件存在检查

### Claude's Discretion
- 具体的请求间隔数值 (2-3 秒 + 抖动) — 研究时根据 pixiv API 特性调整
- 标题特殊字符清理规则 — 确保 Windows 兼容,不破坏文件名可读性
- JSON 输出的详细字段设计 — 包含足够信息供第三方使用

</decisions>

<specifics>
## Specific Ideas

- **第三方调用优先**: 所有设计决策优先考虑第三方程序调用的便利性
  - 默认 JSON 输出
  - 清晰的错误信息
  - 退出码表示成功/失败
- **保守的速率控制**: 宁可慢一些也要稳定,避免触发 pixiv 限制
- **通用文件组织原则**: 为未来扩展 (作品集、画师下载) 预留一致的目录结构模式

</specifics>
